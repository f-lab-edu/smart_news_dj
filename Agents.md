# Agents.md
---

## 🎯 시스템 목표

* 추천 서버 없이 온디바이스에서 개인화 추천 수행
* 사용자 행동 데이터를 기반으로 관심사 학습
* 단순하지만 확장 가능한 구조 유지

---

## 🏗 전체 구조

```
UI (Compose)
    ↓
ViewModel (MVVM, StateFlow)
    ↓
UseCase
    ↓
Recommender (핵심 로직)
    ↓
Repository
    ↓
API / DB
```

---

## ⚙️ 핵심 설계 원칙

### 1. 추천 로직은 Domain에 격리

* 추천 알고리즘은 UI / Data 레이어와 분리
* 테스트 가능하고 독립적인 구조 유지

---

### 2. 단일 데이터 스트림 유지

* API는 단일 쿼리 ("뉴스") 사용
* 페이징 안정성 확보
* 다중 쿼리 사용 금지

---

### 3. 카테고리 사용하지 않음

* 외부 API에서 카테고리 제공 없음
* 대신 키워드 기반 유사도 사용

---

### 4. 키워드 기반 개인화

* 기사 → 키워드 추출
* 사용자 행동 → 키워드 점수 반영
* Top-K 유지 (최대 20개)

---

### 5. 노이즈 제어 전략

* 기사당 키워드 최대 5개 제한
* 불용어 제거
* 조사 제거 (normalize)
* Top-K 유지로 자동 정리

---

### 6. 시간 기반 관심사 변화 반영

* decay 적용
* 오래된 키워드 점수 감소

---

## 🧠 추천 시스템 구성

### 1. KeywordExtractor

* title + description 기반 키워드 추출
* 공백 기준 split

---

### 2. KeywordNormalizer

* 조사 제거 (예: "삼성이" → "삼성")
* 특수문자 제거
* 소문자 변환

---

### 3. UserProfileManager

* 키워드별 점수 관리
* Top-K 유지 (최대 20개)

decay 적용

---

### 4. RecommendationEngine (핵심)

```text
입력:
- 기사 리스트
- 유저 프로파일

출력:
- 추천 점수 기반 정렬된 기사 리스트
```

---

## 📊 사용자 행동 해석

### 수집 데이터

* 클릭 (상세 진입)
* 체류시간 (핵심)
* 스크롤 70% 이상

---

### 점수 설계

```
score = click + dwellTime + scroll
```

---

### 신호 강도

* Strong: dwellTime
* Medium: scroll
* Weak: click

---

## 🔄 추천 생성 방식

* 총 20개
* 70% 개인화 + 30% 랜덤

---

## 🔄 추천 갱신 시점

* 앱 진입 시
* 사용자 새로고침
* 사용자 행동 발생 시 (권장)

---

## 📦 레이어별 책임

### Presentation (MVVM)

* UI 상태 관리 (StateFlow)
* 사용자 이벤트 처리
* UseCase 호출

---

### Domain

* 추천 로직
* 키워드 처리
* 점수 계산
* 유저 프로파일 관리

---

### Data

* API 호출 (네이버 뉴스)
* 로컬 DB 저장 (Room)
* Repository 구현

---

## ⚠️ 주의 사항

* ViewModel에 추천 로직을 넣지 않는다
* 키워드 정확도보다 점수 설계가 더 중요
* 키워드는 noisy하다는 전제로 설계

---

## 🚀 확장 방향

* 동의어 매핑 추가
* 키워드 clustering
* embedding 기반 추천
* 서버 기반 추천 시스템으로 확장
